# Nook research: what to learn for Opencaster

Source inspected: <https://github.com/wojventures/nook>

Inspection date: 2026-06-03

Inspected revision: `aba8569`

Our client: no-build, read-only Express/EJS Farcaster client.

## TL;DR

Nook is valuable as a map of what a serious Farcaster client eventually needs: hydrated cast cards, profile/channel context, route-backed tabs, media/frame display modes, batch hydration, cursor feeds, content metadata, denormalized protocol tables, and event-driven stats.

Do **not** copy Nook source directly. The repository has no explicit license file or package license, only a README note that the code is provided “as-is for people to learn from and build upon.” Treat it as idea/reference material, not reusable code. Also, Nook was a full write-capable, multi-service production system; most of that is wrong-sized for Opencaster right now.

Best immediate takeaways for us:

1. Improve cast rendering: byte-offset-safe mentions/channels, embed URL hiding, richer media/quote cards.
2. Add route-backed modes: home/recent/media/frames, profile casts/replies/media, channel relevant/latest/media.
3. Keep service/view separation: routes ask provider/service for hydrated view models; EJS never knows provider raw payloads.
4. Add batch hydration discipline: page cast hashes first, then fetch related users/casts/stats/embeds together.
5. Document a future owned-index schema around casts/users/reactions/links/content references, but do not build it until the provider path is exhausted.

## Nook architecture at a glance

Nook is a TypeScript/Yarn workspaces monorepo:

```txt
apps/
  expo     native app
  next     web app
packages/
  api                 main API
  app                 shared app features
  app-ui              shared UI primitives
  common              shared types, Prisma clients, queues, cache/API clients
  content-api         URL/embed/frame metadata
  dashboard           queue visualization
  events              post-process Farcaster events
  farcaster           hub publisher/consumer/backfill
  farcaster-api       Farcaster read API
  list-api            lists
  notifications       push notification worker
  notifications-api   notification API
  scheduler           scheduled casts cron/worker
  signer-api          signer custody + Farcaster writes
  swap-api            0x swap API, apparently unused
```

Stack from README:

- TypeScript
- BullMQ / Redis queues
- Postgres / Prisma
- Redis cache
- Fastify APIs
- Tamagui / Expo Router / Next / React Query / Zustand

This is a serious production social-client stack, but it is intentionally the opposite of Opencaster's current value: small, read-only, server-rendered, no frontend build.

Adaptation for Opencaster:

### 5. Byte-offset-safe cast text rendering

Reference file:

- `packages/app/components/farcaster/casts/cast-text.tsx`

This is one of the most important implementation lessons.

Nook handles mention/channel positions with a UTF-8 `Buffer`, sorts mentions by descending byte position, linkifies URLs, and suppresses URLs already represented as embeds.

Adaptation for Opencaster:

- Implement a server-side safe text renderer that:
  - escapes all user text
  - slices by UTF-8 byte positions, not JS string indices
  - converts user mentions to `/u/:username` or `/fid/:fid`
  - converts channel mentions to channel/feed routes when available
  - linkifies safe `http(s)` URLs
  - hides plain URLs already rendered as embed cards
- Add tests with emoji/multibyte text before and around mentions. Naive `text.slice(position)` breaks here.

Avoid:

- Raw HTML injection from cast text.
- Character-index slicing for protocol-provided byte offsets.

## Backend/data patterns worth adapting later

### 1. Separate provider/raw data from hydrated display models

Reference file:

- `packages/farcaster-api/src/service/farcaster.ts`

Nook's read service does the important thing: it starts from cast hashes/base objects, fetches related users/casts/context/embeds in batches, then assembles `FarcasterCastV1` display objects.

Adaptation for Opencaster:

- Keep provider adapters narrow.
- Add a hydration layer that returns stable app view models.
- Batch related users/casts/embeds for each page before rendering.
- EJS should render prepared cards, not query or understand provider payload shapes.

Avoid:

- A huge 2,000-line service file. Split by casts/users/feeds/embeds as we grow.

### 2. Cursor feed model

Reference files:

- `packages/common/types/feed.ts`
- `packages/app/features/farcaster/cast-feed/filtered-feed-server.tsx`
- `packages/app/features/farcaster/cast-feed/infinite-feed.tsx`
- `packages/app/hooks/api/feed.ts`

Nook feed requests support channels, users, text, embeds, content types, replies, frames, cursor, and limit.

Adaptation for Opencaster:

- Keep our feed presets, but formalize a filter object:
  - `query`
  - `channel`
  - `fid`
  - `includeReplies`
  - `onlyReplies`
  - `onlyMedia`
  - `onlyFrames`
  - `cursor`
  - `limit`
- Use base64 JSON cursors when provider cursors are not directly reusable.

Avoid:

- Offset pagination and repeated page-one bugs.
- Filters that silently fall back to unrelated content.

### 3. Owned-index schema, if/when we outgrow managed providers

Reference file:

- `packages/common/prisma/schemas/farcaster.prisma`

Nook's Farcaster tables model the protocol well:

- `FarcasterCast`
- `FarcasterCastMention`
- `FarcasterCastEmbedCast`
- `FarcasterCastEmbedUrl`
- `FarcasterCastReaction`
- `FarcasterUrlReaction`
- `FarcasterLink`
- `FarcasterUserData`
- `FarcasterVerification`
- `FarcasterUsernameProof`
- `FarcasterParentUrl`
- stats tables for casts/users/parent URLs
- `Subscription` cursor for hub events

Patterns to adapt later:

- Soft deletes with `deletedAt`.
- Protocol-natural compound IDs, e.g. reaction `[targetHash, reactionType, fid]` and follow `[fid, linkType, targetFid]`.
- Denormalized parent/root/channel fields on casts.
- Separate mention/embed edge tables.
- Materialized counters for fast reads.
- Indexes aligned to route reads: author+timestamp, parent hash, root parent URL/channel, timestamp.

Small future schema shape for us:

```txt
casts(hash, fid, text, timestamp, parent_hash, parent_fid, parent_url,
      root_parent_hash, root_parent_url, deleted_at, raw_json)
users(fid, username, display_name, pfp, bio, updated_at, raw_json)
reactions(target_hash, reaction_type, fid, timestamp, deleted_at)
links(fid, target_fid, link_type, timestamp, deleted_at)
cast_stats(hash, likes, recasts, replies, quotes)
content_refs(hash, uri, content_type, metadata_json, has_frame, updated_at)
```

Do not build this now unless the product goal becomes owned search/ranking/analytics. Current client should stay provider-backed.

### 4. Event ingest and backfill pattern

Reference files:

- `packages/farcaster/src/publisher.ts`
- `packages/farcaster/src/consumer.ts`
- `packages/farcaster/src/processor/index.ts`
- `packages/farcaster/backfill/processor.ts`
- `packages/common/farcaster/message.ts`
- `packages/common/farcaster/events.ts`

Nook pipeline:

1. Subscribe to Farcaster Hub `MERGE_MESSAGE` events.
2. Resume from stored subscription event ID.
3. Push each message into BullMQ with deterministic job ID.
4. Process by `MessageType`.
5. Upsert protocol rows and soft-delete removals.
6. Update stats.
7. Publish app-level events.

Patterns to adapt later:

- Store last processed event ID.
- Idempotent writes via natural keys/upserts.
- Use the same converters for live ingest and backfill.
- Derive app-level events after durable writes.
- Update stats at ingest time.

Avoid now:

- BullMQ/Redis and multi-worker complexity.
- Global full-Farcaster backfill.
- Non-transactional stats updates if we build our own index; wrap write+stats in a transaction.

### 5. Content/embed metadata cache

Reference files:

- `packages/content-api/src/service/content.ts`
- `packages/content-api/src/utils/url.ts`
- `packages/common/prisma/schemas/content.prisma`
- `packages/common/clients/cache/content.ts`

Nook treats URL metadata as its own system:

- content reference records keyed by `uri + fid + hash + type`
- parsed URL components
- content type and length
- metadata JSON
- frame JSON and `hasFrame`
- cache-aside Redis reads
- lazy/async fetching option

Adaptation for Opencaster:

- Add a small URL metadata cache only when embed cards become a quality bottleneck.
- Fetch lazily with strict timeouts.
- Store host/title/description/image/contentType per URL.
- Never block a feed page on many remote scrapes.

Avoid:

- SSRF-prone arbitrary URL fetching.
- Synchronous scraping of many URLs during render.
- Saving Cloudflare/error pages as valid metadata.

- `packages/signer-api/src/routes/signer.ts`

- `packages/common/prisma/schemas/signer.prisma`

Do not copy:

- signer custody
- stored Farcaster signer private keys
- cast/like/recast/follow submission endpoints
- Privy/session complexity

Reason: Opencaster is read-only. Writes need a separate threat model, SIWF/auth, managed signer strategy, CSRF/rate limits, and user confirmation.

Security notes from inspection:

- Many routes appear to use JWT decode patterns for auth-sensitive operations; decode is not verification.
- Signer private keys are DB fields in the signer schema.
- Some signer/scheduled paths look under-protected for our standards.

### Now: no-build client improvements

1. Safe rich cast text renderer
   - byte-offset mention/channel slicing
   - escaped text
   - linkified safe URLs
   - hide URLs already shown as embeds
   - emoji/multibyte tests

2. Central embed view model
   - image/video URL cards
   - quote-cast cards
   - static frame previews
   - host labels
   - strict URL validation

3. Route-backed display modes
   - `/feed/:feedId/media`
   - `/feed/:feedId/frames`
   - `/u/:username/replies`
   - `/u/:username/media`
   - channel routes only if provider can support them cleanly

4. Better thread detail
   - ancestors
   - selected cast expanded
   - direct replies
   - exact timestamp
   - channel/sidebar context

5. Batch hydration cleanup
   - make route handlers request hydrated page view models
   - keep provider raw data behind provider modules
   - prevent per-card fetches

### Next: product quality

1. Search tabs for casts/users/channels.
2. Minimal route-specific metadata/OG tags.
3. Right sidebar context cards.
4. Static list/feed preset pages using config, not DB/auth.
5. Optional PWA manifest.

### Later: owned data plane

Only if provider-backed UX hits a ceiling:

1. SQLite/Postgres mini-index for viewed feeds/casts.
2. URL metadata cache with SSRF-safe fetcher.
3. Hub/Snapchain ingest cursor and idempotent processing.
4. Materialized stats/ranking tables.
5. Saved searches/alerts/agent-readable APIs.

## Why this matters

Nook proves the hard part of a Farcaster client is not rendering a timeline. The hard part is hydration: turning protocol messages, users, channels, embeds, stats, context, and threads into one stable display object without leaking provider complexity into the UI.

That is the part we should copy conceptually.

The rest — signer custody, scheduled posting, push notifications, swaps, many databases, many queues — is a beautiful way to accidentally build a haunted mansion when we currently need a sharp little reader.
